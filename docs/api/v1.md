# History Wall REST API v1

Base path: `/api/v1`

The API stores the canonical civilization, person, event, and era records defined in
`src/contracts/history-wall.schema.ts`. Record identity and historical fields
are append-only; v1 permits authenticated edits to author notes/details, but no
PUT, DELETE, or upsert behaviour.

All responses use a versioned JSON envelope:

```json
{
  "apiVersion": "v1",
  "data": {}
}
```

Errors use `{ "apiVersion": "v1", "error": { "code", "message" } }`.

## List every record

```http
GET /api/v1/records
```

Returns summaries for every civilization, person, event, and era, sorted by start year.
Each summary contains the stable ID, record type, title, start/end years, optional
civilization link, creation time, and last note-update time. Multi-civilization interaction summaries
also include `civilizationIds` and `interactionType`. Large notes, geography, color, icons,
visuals, metadata, values, and tags are omitted.

To explicitly request every full record:

```http
GET /api/v1/records?detail=full
```

The list envelope also includes `detail` and `count`.

## Get one full record

```http
GET /api/v1/records/:id
```

Returns the complete canonical JSON payload plus its server-assigned
`createdAt` and `updatedAt`. Missing IDs return `404`.

## Add one record

```http
POST /api/v1/records
Authorization: Bearer $HISTORY_WALL_WRITE_SECRET
Content-Type: application/json
```

The body is one `civilization`, `person`, `event`, or `era` object—not the
top-level base JSON containing four arrays. For example:

```json
{
  "type": "event",
  "id": "event_vesuvius_79",
  "title": "Eruption of Vesuvius",
  "span": {
    "startYear": 79,
    "displayLabel": "79 CE",
    "certainty": "exact"
  },
  "civilizationId": "civilization_rome",
  "notes": "Vesuvius buried Pompeii and nearby settlements.",
  "metadata": {}
}
```

Civilizations must be added before people, events, or eras that reference them. A valid
insert returns `201` and a `Location` header. Invalid records/references return
`422`; a reused ID returns `409` and never overwrites the existing record.

Record write bodies are limited to 128 KiB. The `metadata` object is limited to
100 top-level keys and 32 KiB of JSON.

## Edit author notes

```http
PATCH /api/v1/records/:id
Authorization: Bearer $HISTORY_WALL_WRITE_SECRET
Content-Type: application/json
```

The body may contain `notes`, `details`, or both. Identity, type, dates, title,
and civilization relationships cannot be changed. Set `details` to `null` to
remove the structured details object. The response includes the original
`createdAt` and a new `updatedAt`.

```json
{
  "notes": "Updated short summary.",
  "details": {
    "markdown": "Long-form study note.",
    "tags": ["rome"],
    "sources": [],
    "media": []
  }
}
```

### Add a person

```json
{
  "type": "person",
  "id": "person_hatshepsut",
  "title": "Hatshepsut",
  "span": {
    "startYear": -1507,
    "endYear": -1458,
    "displayLabel": "c. 1507–1458 BCE",
    "certainty": "approximate"
  },
  "civilizationIds": ["civilization_egypt"],
  "roles": ["monarch", "political_leader"],
  "roleTitle": "Pharaoh",
  "notes": "An Eighteenth Dynasty pharaoh.",
  "metadata": {}
}
```

`civilizationIds` may be empty or contain several existing civilizations.
`span` is normally the person's lifespan; use the best-known active span when
birth or death dates are not known.

### Add a multi-civilization interaction

An interaction is one event with two or more participants. Use this for wars,
trade, exploration/contact, alliances, conquest, colonialism, hegemony, and
other cross-civilization relationships:

```json
{
  "type": "event",
  "id": "event_example_war",
  "title": "Example multi-party war",
  "span": {
    "startYear": 1939,
    "endYear": 1945,
    "displayLabel": "1939–1945 CE",
    "certainty": "exact"
  },
  "interaction": {
    "type": "war",
    "participants": [
      { "civilizationId": "civilization_a", "role": "ally" },
      { "civilizationId": "civilization_b", "role": "aggressor" }
    ]
  },
  "notes": "Short summary for lists and timeline labels.",
  "metadata": {}
}
```

Do not also provide `civilizationId` on an interaction event. Every participant
must already exist as a civilization record.

## Generate an event image

```http
POST /api/v1/images/generate
Authorization: Bearer $IMAGE_GENERATION_SECRET
Content-Type: application/json
```

```json
{
  "title": "The Battle of Hastings",
  "description": "Norman cavalry and English infantry meet near Hastings in 1066 CE.",
  "provider": "auto",
  "aspectRatio": "16:9",
  "style": "editorial"
}
```

`provider` may be `auto`, `openai`, or `gemini`. `auto` uses `IMAGE_PROVIDER`,
then an available key. The response includes provider, model, MIME type, and a
base64 data URL for immediate preview. This endpoint generates but does not
permanently host the image.

## Storage model

PostgreSQL table `history_records` stores the full validated record in a JSONB
`payload`. Stable ID, type, title, start/end years, civilization reference,
schema version, and creation time are also stored as typed columns for fast list
queries and database constraints.

Apply the schema to the configured Neon database with:

```bash
pnpm db:push
```
